## Introduction

Distributed machine learning is increasingly being used in ride-sharing apps to enable more efficient and accurate data processing. In this approach, the data collected from users is split into smaller subsets or partitions and distributed across multiple servers or nodes. Machine learning algorithms are then applied to these subsets, and the resulting models are combined to generate predictions and recommendations. This approach allows for faster processing of large datasets, as well as improved accuracy due to the use of more diverse data. Additionally, distributed machine learning can help ensure user privacy by limiting access to sensitive data. 

## What do they use ML for?

Ride-sharing apps use machine learning algorithms to optimize their pricing, calculate estimated arrival times, and match riders with drivers based on location and demand. These algorithms continuously learn from the vast amounts of data generated by the app's users, improving the accuracy of their predictions over time. It is also used to identify fraudulent behavior, such as fake accounts or credit card fraud, and to ensure the safety of riders and drivers. The primary use-cases are:

- Price Optimisation
- Safety
- ETAs
- Fraud Detection
- Mapping
- Incentives

## What are the technical scenarios for distributed ML?

Some of the technical challenges that pop-up during distributed ML application:

- Feature Engineering on petabyte scale datasets.
- Building ML pipelines using the largest machine on AWS?
- How to harmoniously use 1000 GPUs for training a model?
- What is the best combination out of the 1 Billion hyperparameters?

These are just the edge cases, every stage in the ML Lifecycle involves distributed computing of some sort. Starting from Feature Engineering to Prototyping to Model Training to Serving and finally setting up monitoring and observability practices.

## What are the best design principles in distributed computing when it comes to ride-sharing apps?

- Divide the data in smaller chunks
- Lifecycle iteration should be fast
- Freedom to update to the latest libraries
- De-coupled systems
- Time-cost equilibrium should be taken care of
- Ease of use and maintenance

## Case Study

Let us take the example of Lyft which is a famous ride-sharing app and a direct competitor to Uber and see how they deploy federated and distributed systems in ML:

### Architecture of Distributed Environment

The primary distributed workflows used by Lyft consists of three types: Computing, Training and Tuning. Libraries like Lyft Engines, Spark, Presto, Hive, Pandas, Fugue, Ray, Lyft Mozart are used as computation and tuning engines. For training, Tensorflow, Pytorch, Horovod, Kubeflow etc are used. Specialised libraries like Optuna and Hyperopt are used for tuning.

### Spark and Deep learning on Kubernetes

Spark is a framework used for running distributed applications efficiently on a cluster of compute instances which in our case is a Kubernetes cluster. Spark follows a driver-worker architecture where the driver in association with a resource manager like YARN, Apache Mesos etc divides the application/job into stages and the stages into tasks and assigns them to the workers. The workers do the distributed computation. The driver is launched on the client side job launching environment or among the clusters based on whether the mode is client-mode or cluster-mode.

<img src="/assets/img/rs_dist_3.png" alt="distributed environment">

<img src="/assets/img/rs_dist_4.png" alt="distributed environment">

 Training deep learning models is done on Kubernetes clusters. In deep learning we need to calculate the gradient in order to be able to adjust our weights. Without this learning can’t happen. In order to calculate that gradient, we need to process all of the data. When such data is too big this becomes a problem. This is why we parallelize these calculations. This means that we will have several computers working in parallel on a subset of the data. When each of these processing units or workers (they could be CPUs, GPUs, TPUs, etc.) is done calculating the gradient for its subset, they then need to communicate its results to the rest of the processes involved. Actually, every process needs to communicate its results with every other process/worker.Fortunately, someone else had that problem in the past and devised the Allreduce and Ring all-reduce algorithm, which does precisely this. The Allreduce algorithm has every process/worker share its data with all other processes and applies a reduction operation. Uber Horovod is used for this purpose.

### Example implementation in Pyspark:

The following function yields a generator function that produces a sequence of dictionaries. Each dictionary corresponds to a task and contains a **`result`** key with the computed result of that task. The use of a generator allows the function to lazily produce results one at a time, rather than generating all of the results at once and storing them in memory, which can be more memory-efficient for large datasets.

```python
def compute_tasks (tasks :pd.DataFrame) -> Iterable[Dict[str,Any]]:
	for task in tasks.to_dict( "records"):
	 	#some complex or expensive computation
		task["result"] == str(task)
		yield task
```

Now we’ll create a k8s_spark session using credentials verified distributed library (xyz_distributed) and apply the `fugue.transform` using spark as compute engine. 

```python
from xyz_distributed import k8s_spark, local_spark
from fugue import transform

with k8s_spark({"cluster:"4*4*4g"}) as session:
    sdf = session.createDataFrame([[1,"a"], [2,"b"]], "a:int,b:string")
    result = transform(sdf, compute_tasks, engine="active_sparksession")
    print(type(result))
    result.show()
```

We can also run predictions in a distributed manner using `fugue.transform`

```python
from fugue import transform

def predict(df: pd.DataFrame) -> pd.DataFrame:
     region = df.region.iloc[0]
     model = load_model(region)
     return df.assign(pred = model.predict(df))

all_region_data = spark_session.read.parquet("<s3 location>")
result = transform(
	all_region_data, # data
	predict, # logic
	partition={"by": "region"}, # logical partition
	engine= spark_session # run on current spark session
)
result.write.parquet("<s3 location>")
```

## Challenges in Distributed Training

- Having a standalone cluster is expensive. Spark on K8s blends in well.
- Users want their custom docker images as Spark executor. K8s helps solve this
- Cost has to be kept down by preventing runaway and unoptimised clusters. Using K8s namespaces to isolate users, custom labels and annotations on pods to detect spark executor pods, deploying K8s event listeners to track spark pod events, ensuring graceful shutdown of sparksession using python SDK context managers etc helps mitigate the cost issue to a great extent.
- Job Starvation: Spawning 100s of spark jobs (1000s of executor pods) from one namespace in loop causes the K8s API to choke causing DDOS for other namespaces thus increasing downtime of the platform. Resource quota per namespace should be assigned and backup K8s job controllers using custom labels should be deployed on spark executors to destroy orphan pods.
